### Q1: 在 Low Code 平台的開發應用設計中，關於 「模型 （Model） 」 ，下列敘述 何者最符合實際情況？

- (A) 模型僅扮演設計視覺化的輔助工具，對應用邏輯的影響有限；
- <span style='color: #0066cc; font-weight: bold;'>(B) 模型是用來抽象描述資料結構、業務流程與介面邏輯的核心元素， 影響應用的設計與維護；</span>
- (C) 模型僅依循 UML（Unified Modeling Language）等傳統建模方式， 缺乏針對 Low Code 環境的延展性；
- (D) 模型在 Low Code 平台中已被自動程式碼生成全面取代，實際價值 有限

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(B) 模型是用來抽象描述資料結構、業務流程與介面邏輯的核心元素， 影響應用的設計與維護；</span>  
<span style='color: #646464;'>💡 為什麼？ 在 Low Code (低程式碼) 平台中，「模型驅動 (Model-Driven)」是核心概念。開發者建立資料模型、流程模型與介面模型來「宣告式」定義應用的行爲，平台再自動將這些抽象的視覺化模型轉換為底層程式碼。它是應用程式的設計骨幹，不僅限於視覺輔助。</span>

### Q2: 企業在導入生成式 AI 平台時，往往需要利用分散於不同部門或機構中 的大量敏感文本資料 。 若希望在確保隱私的前提下 ， 仍能讓模型持續優 化並降低資料外洩風險，下列哪一種方法最適合？

- (A) 同態加密（Homomorphic Encryption）；
- (B) 安全多方計算（Secure Multi-party Computation）；
- (C) 零知識證明（Zero-knowledge Proofs）；
- <span style='color: #0066cc; font-weight: bold;'>(D) 聯邦學習（Federated Learning）</span>

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(D) 聯邦學習（Federated Learning）</span>  
<span style='color: #646464;'>💡 為什麼？ 聯邦學習 (Federated Learning) 讓模型直接在各部門或機構的本地端進行訓練，只將不包含原始敏感資料的「模型權重更新 (Gradients)」傳送到中央伺服器進行聚合。這完美達成了「資料不動、模型動」的最高隱私保護境界。</span>

### Q3: 某企業利用 No Code/Low Code 平台開發內部營運系統。為確保系統在 跨部門流程與外部服務整合下仍具良好的可測試性 （Testability） ， 下 列哪一項作法最為合適？

- (A) 依賴 No Code/Low Code平台提供的即時預覽與基本單元測試功能 ， 快速驗證常見流程；
- <span style='color: #0066cc; font-weight: bold;'>(B) 導入可重複執行的自動化測試流程，並透過 API 或服務虛擬化進 行模組化驗證；</span>
- (C) 將測試聚焦於使用者介面互動與操作流程驗證，檢查系統表面功 能；
- (D) 依靠使用者回饋與正式上線後的監控資料，作為修正依據

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(B) 導入可重複執行的自動化測試流程，並透過 API 或服務虛擬化進 行模組化驗證；</span>  
<span style='color: #646464;'>💡 為什麼？ 在整合外部服務或跨部門流程時，無法僅依賴平台內建的單機即時預覽功能。(B) 透過建立自動化測試與服務虛擬化 (Mocking/Stubbing API)，能在不影響真實生產環境的前提下，穩定、可重複地驗證各模組間的資料拋接與邏輯正確性。</span>

### Q4: 某社交平台嘗試結合自動提示工程（ Automatic Prompt Engineer, APE） 與圖提示 （Graph Prompting） ，讓AI 協助分析使用者之間的互動 關係。在這個過程中，下列何者為最可能遇到的挑戰？

- <span style='color: #0066cc; font-weight: bold;'>(A) 圖結構轉換為文字提示時，可能導致部分關聯資訊遺失；</span>
- (B) APE 在圖資料上無法產生任何提示內容；
- (C) 圖轉文字後能完整保留所有上下文，對推理不造成影響；
- (D) Graph Prompting 僅能處理線性路徑，限制多分支探索

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(A) 圖結構轉換為文字提示時，可能導致部分關聯資訊遺失；</span>  
<span style='color: #646464;'>💡 為什麼？ 圖資料 (Graph Data) 本身具有複雜的拓樸網絡與非線性的高維節點關聯。當我們強制將這些圖結構攤平、降維轉換成一維序列的「文字提示 (Prompt)」餵給語言模型時，非常容易流失其原有的空間結構與深層的相互依賴關係。</span>

### Q5: 在超長上下文任務中使用自動提示工程 （Automatic Prompt Engineer, APE）時，可能面臨的最大限制是什麼？

- (A) 迭代優化難以因應上下文的不斷變動，導致調整失效；
- (B) 模型的記憶容量有限，無法完整保留所有長篇資訊；
- (C) 提示內容難以有效分解，無法支援複雜任務拆解；
- <span style='color: #0066cc; font-weight: bold;'>(D) 回饋機制通常僅針對局部片段，難以全面評估最終輸出品質</span>

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(D) 回饋機制通常僅針對局部片段，難以全面評估最終輸出品質</span>  
<span style='color: #646464;'>💡 為什麼？ 當處理數萬字以上的超長上下文任務時，APE 生成的新 Prompt 往往只能透過片面的「短文本採樣」來快速評估效果。由於缺乏對超長文本全局一致性的快速、精確衡量評分系統，APE 很難找出能完美兼顧「頭尾全局上下文」的最佳提示詞。</span>

### Q7: 某團隊希望讓 AI 自動查詢 GitHub 上的程式碼庫，並生成摘要給使用 者參考。開發者決定透過Model Context Protocol（MCP）來實現，AI 需先發出請求，再經由 MCP 架構逐步完成查詢與回傳。在此情境下， MCP 運作流程的正確順序為何？

- (A) MCP Server →AI Host →MCP Client →資料查詢→結果回傳 AI Host；
- (B) MCP Client →AI Host →MCP Server →資料查詢→結果回傳 AI Host；
- <span style='color: #0066cc; font-weight: bold;'>(C) AI Host →MCP Client → MCP Server →資料查詢→結果回傳 AI Host；</span>
- (D) AI Host→MCP Server→MCP Client→資料查詢→結果回傳 AI Host

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(C) AI Host →MCP Client → MCP Server →資料查詢→結果回傳 AI Host；</span>  
<span style='color: #646464;'>💡 為什麼？ MCP (Model Context Protocol) 的標準架構中，AI 主體 (AI Host) 遇到需要外部資訊時，會指示其附屬的 MCP Client 發出請求；Client 再透過標準協定與資料源端的 MCP Server 溝通，伺服器執行查詢後將資料原路返回給 AI 進行後續歸納與推理。</span>

### Q8: 在 Agent-to-Agent（A2A）架構中，不同代理人之間會分工合作。一般 而言，下列敘述何者最符合 Client Agent 與 Remote Agent 的互動流 程？

- (A) Remote Agent 主動分派任務給Client Agent；
- (B) 由人工事先設定 Client Agent 與 Remote Agent 處理任務的先後順 序；
- <span style='color: #0066cc; font-weight: bold;'>(C) Client Agent 發起任務，Remote Agent 執行並回傳結果；</span>
- (D) Client Agent 與 Remote Agent 同時處理任務並同步處理結果

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(C) Client Agent 發起任務，Remote Agent 執行並回傳結果；</span>  
<span style='color: #646464;'>💡 為什麼？ 在多代理人協作 (Multi-Agent System) 的典型主從 (Client-Server/Remote) 架構中，Client Agent 扮演調度與規劃大腦，由它向專門負責特定執行任務的 Remote Agent 發出指令 (Request)，Remote Agent 完成運算或動作後再將成果交還給 Client。</span>

- (A) 縮短模型訓練時間；
- <span style='color: #0066cc; font-weight: bold;'>(B) 優化提示與上下文；</span>

- (D) 優化 Fine-tuning 正確率

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(B) 優化提示與上下文；</span>  
<span style='color: #646464;'>💡 為什麼？ 上下文工程 (Context Engineering) 無關乎更改模型本身的權重參數 (如模型微調)，它的核心在於研究如何有效地檢索、過濾、組織並提供最精準的「外部背景知識 (Context)」與「提示詞 (Prompt)」餵給模型，以大幅降低幻覺並提升回答精準度。</span>

### Q10: 某公司在導入生成式 AI 協助撰寫內部報告時，測試人員刻意在輸入的 上下文中放入互相矛盾的資訊 （例如 ： 同一位員工在不同段落被描述為 「入職三年」 與 「入職五年」 ） 。在這種情況下，最常見的模型行為會是 什麼？

- (A) 永遠選擇第一段資訊作為答案依據；
- <span style='color: #0066cc; font-weight: bold;'>(B) 可能生成幻覺或隨機採信其中一方的內容；</span>
- (C) 拒絕回答，並要求提供更一致的輸入；
- (D) 自動判斷並只選擇正確的資訊

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(B) 可能生成幻覺或隨機採信其中一方的內容；</span>  
<span style='color: #646464;'>💡 為什麼？ 當 Prompt 輸入中包含矛盾的上下文時，大型語言模型 (LLM) 缺乏像人類一樣主動發問澄清的直覺機制。它會試圖在矛盾的資料分布中尋找最可能的接續文字，這往往會導致生成出強行縫合兩邊資訊的「幻覺 (Hallucination)」，或是隨機倒向採信其中一段資料。</span>

### Q11: Agentic AI 在解決方案圖譜（Solution Graph）上尋找最佳解決路徑 時，通常會使用什麼樣的搜尋策略？

- <span style='color: #0066cc; font-weight: bold;'>(A) 使用廣度優先、深度優先或最佳優先等演算法進行探索；</span>
- (B) 每一步都隨機選擇動作，反覆嘗試直到找到一條可行路徑；
- (C) 只執行事先假定的一條路徑，失敗就停止；
- (D) 完全依靠 LLM 一次性推斷最優完整路徑

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(A) 使用廣度優先、深度優先或最佳優先等演算法進行探索；</span>  
<span style='color: #646464;'>💡 為什麼？ Agentic AI (自主代理 AI) 的核心能力在於「規劃 (Planning)」與「搜尋 (Search)」。當面對解空間呈樹狀結構 (Solution Graph) 時，它會採用經典演算法（如 BFS、DFS 或 A* 最佳優先搜尋）結合啟發式函數，有條理地探索並推斷出最佳行動路徑，而非盲目隨機或瞎猜。</span>

### Q12: 某企業考慮將開源大型語言模型（GPT-OSS）自行部署在本地伺服器， 以取代雲端服務。下列何者最能代表本地部署對企業的實際好處？

- (D) 以上皆是

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(C) 可確保輸入模型的敏感資料不會傳輸給第三方，提升資料隱私和自 主控制；</span>

### Q15: 某客服自動回應系統希望根據不同客戶群體調整回覆風格。在兼顧即 時性與效果的前提下，下列哪一種方案最適合？

- (A) 直接微調預訓練模型針對每個客戶群體分別訓練不同風格模型；
- <span style='color: #0066cc; font-weight: bold;'>(B) 利用控制變量（Control Tokens）或風格標籤在同一模型內動態調 整風格；</span>

### Q43: 在評估大型語言模型，例如7B、13B、175B 參數規模時，模型規模對基 準測試（Benchmark）結果的影響，下列哪一種說法最為恰當？

- (A) 小模型在正確調整下能超越大模型，因此模型大小並不重要；
- (B) 大模型在多數情境下表現較好，但在特定任務上略遜於小模型；
- (C) 模型規模與基準測試結果完全無關，影響主要來自測試設計；
- <span style='color: #0066cc; font-weight: bold;'>(D) 模型越大，Benchmark 結果可能提升，但幅度取決於訓練數據品質 與資源配置</span>

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(D) 模型越大，Benchmark 結果可能提升，但幅度取決於訓練數據品質 與資源配置</span>

### Q47: 某顧問公司導入生成式 AI， 協助團隊快速檢索並摘要長篇的法規文件 。 為了改善檢索結果常出現不相關或過於分散內容的問題，下列何者為 團隊決定對文件進行文本切分（Chunking）的主要目的？

- (A) 讓模型在回答時能加快推理速度；
- <span style='color: #0066cc; font-weight: bold;'>(B) 提高檢索相關性與降低長上下文噪音；</span>

- (D) 使模型在生成回覆時更具創造性與多樣化

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(B) 提高檢索相關性與降低長上下文噪音；</span>

### Q48: 某開發團隊在建置企業內部知識檢索系統時，選擇採用多向量檢索器 （Multi-vector Retriever） ，下列何者為協助提升系統查詢的完整性 與精準度的主要方式？

- <span style='color: #0066cc; font-weight: bold;'>(A) 支援同時處理多種資訊表示，提升跨文本型態的檢索效果；</span>

- (C) 以切分並過濾文件片段，減少上下文長度帶來的 Token 負擔；
- (D) 透過調整生成階段的溫度參數，使模型在回覆時更穩定一致

<span style='color: #008000; font-weight: bold;'>✅ 正確解答：(A) 支援同時處理多種資訊表示，提升跨文本型態的檢索效果；</span>  
<span style='color: #646464;'>💡 為什麼？ 傳統檢索器是一對一（一段原文字轉換一個向量）。多向量檢索器 (Multi-vector Retriever) 則允許為「單一份文件」生成並關聯「多種不同維度的查詢向量」：例如將同一段落抽取成「條列式摘要」、「圖像描述」或預先產生的「AI 假想問題」。藉由多重維度的火力覆蓋，不論使用者怎麼問都能精準命中文本。</span>
